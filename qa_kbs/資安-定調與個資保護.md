## 一、資安論述三原則（貫穿所有資安題，先記住）

1. **不主動拿建置平台（WordPress／開源框架等）當資安論述**；被直接問「用什麼建置」再照實答，並立刻拉回「**加固＋安全開發流程**」。平台是事實、不是論點。
2. **資安講「四段生命線」，不要只講兩次掃描**：規劃 → 開發 → 上線前 → 維運。
3. **檢測分兩層、球員不兼裁判**：開發期用工具持續掃、上線前找**獨立第三方**正式滲測。

### 標準講法（四段生命線，可整段背）
> 「資安我們分四段落實：**規劃**做權限分層與資料分級；**開發期**走安全開發流程，用 ZAP、Pentest-Tools 等工具**持續自動掃描**、及早收斂；**上線前**委由**資安共契 A 級的獨立第三方（如中華資安、數聯）執行滲透測試並出具簽署報告**——球員不兼裁判；**維運期**落實即時修補、備份回復、稽核留痕，並依法建立個資外洩通報機制。這份獨立滲測已編列在標價的資安測試項內。」

### 若被問「用某開源平台是否安全」——別踩坑
❌ 不要說「開源所以安全」（外行話、會扣分）。
✅ 改說：
> 「資安不建立在『它是開源』，而是建立在**我們的加固與安全開發流程**：外掛／元件最小化、只用維護中的元件、後台 2FA、權限最小化、WAF；自訂邏輯由我們**自主安全審查**，不依賴社群；並在保固期內落實漏洞追蹤與即時修補。」

## 二、個資保護：開發到交付如何確保「碰不到個資」

### 誠實定調（先講，資安委員最吃這套）
> 「完全不碰個資」不現實——資料移轉、線上事故除錯必然接觸真實資料。正確目標是：**預設接觸不到、必要時最小化＋受控＋留痕**。把「碰不到」講成絕對值，內行人一聽就知道是外行話（同「開源所以安全」的坑）。

### ① 開發過程（核心一句：開發／測試環境根本沒有真實個資）
- **環境三分離（ISO 27001:2022 A.8.31）**：dev／test／prod 隔離；開發與測試資料庫**一律不放正式個資**。
- **測試資料用合成／去識別化資料（A.8.33、A.8.11）**：假資料產生器造結構相同、內容虛構的測試資料；非用不可時先去識別化／假名化／遮罩，**絕不整包複製正式個資到開發環境**。
- **工程師無正式環境權限（A.5.15 最小權限）**：開發帳號無正式資料庫／主機存取權，正式環境由受控部署流程上線，人不直接進去撈資料。
- **個資不落 log／不進版控／不進截圖**：log 遮蔽個資欄位；金鑰連線字串不進 git（`.env`／`.gitignore`）；除錯與教學截圖一律用假資料。
- **設計即保護（Privacy by Design）**：欄位設計時做資料分級與最小蒐集，只收特定目的必要欄位，敏感欄位規劃加密與遮罩。
- **安全開發＋自動掃描**：開發期用 ZAP／Pentest-Tools 持續掃，上線前抓掉越權查詢、SQL injection 撈整表等「個資外洩型漏洞」。

> 開發期一句話：「開發與測試環境用的是合成／去識別化假資料，工程師沒有正式資料庫權限、個資也不落日誌——**寫程式的人從頭到尾碰不到真實個資**。」

### ② 唯一必然接觸真實個資的關卡：資料移轉（主動講反而加分）
- 最小人員＋簽保密切結；在加密、隔離的移轉專用環境作業，**不落地個人電腦**。
- 加密傳輸與儲存（A.8.24）；以欄位對映與批次程式為主、減少人工逐筆讀個資。
- **用後即清＋留痕**：移轉驗證完成後清除工作副本，保留處理紀錄備查。

### ③ 交付之後（維運／保固期）（核心一句：交付即收回開發存取）
- **交付即收回權限**：上線後廠商開發人員存取權回收，正式資料日常存取權在機關手上。
- **維運除錯採 break-glass**：機關核准 → 限時開通 → 全程留痕 → 事後關閉，不留長期後門。
- **個資派送最小化（A.8.11）**：後台只派送必要欄位，一般承辦看到的個資預設遮罩，解遮罩需留痕。
- **稽核留痕不可竄改（A.8.15）**：誰在何時查了哪筆個資可回溯，經得起政風／個資稽核。
- **資料生命週期與退場（A.8.10）**：訂保存期限、到期刪除；退場／換廠商時資料交還機關、廠商端副本清除並出具證明。
- **外洩防護與通報**：DLP（A.8.12）＋個資外洩通報 SOP（見 [[資安-滲透測試與外洩通報]]）。

### ④ ISO 27001:2022 控制項對映（讓「已驗證」變「查得到」）
| 做法 | 控制項 |
|---|---|
| 開發／測試／正式環境分離 | **A.8.31** |
| 測試不用真實個資 | **A.8.33** |
| 個資遮罩、後台打碼 | **A.8.11** |
| 最小權限、存取控制 | **A.5.15** |
| 傳輸／儲存加密 | **A.8.24** |
| 存取個資留痕 | **A.8.15** |
| 到期刪除／退場清除 | **A.8.10** |
| 資料外洩防護 | **A.8.12** |
| PII 隱私保護落地 | **A.5.34** |

個資法對應：特定目的與最小蒐集、個資生命週期管理、安全維護措施、外洩通報，與上表一一呼應。

### ⑤ 備詢殺手鐧（可整段背）
> 「個資保護分開發與交付兩段：**開發期**工程師用合成／去識別化假資料、沒有正式庫權限、個資不落日誌，**寫程式的人碰不到真實個資**；**唯一接觸真實資料的是資料移轉**，走最小人員、加密隔離、用後即清、全程留痕；**交付後**我方存取權即收回，維運進正式環境得經機關核准、限時、留痕，退場時資料交還、副本清除出證明。這些對應我們 ISO 27001:2022 的 A.8.31、8.33、8.11、8.24 等控制項——**已驗證、看得到、查得到**。」

## 三、套用提醒（每案上場前務必核對）
- ⚠️ 以上為業界標準與 ISO 落地做法。上場前釐清**哪些「已在做」、哪些要「為本案補齊」**，別把「應做」講成「已做」被追問破功。
- 有公司級 ISO 27001 驗證者，報得出**控制項編號**公信力最高；把公司級驗證**落地成本案契約承諾**（見 [[採購財法-低價標智財違約與賠償]] C-5 型題）。
- 涉兒少／醫療等**敏感個資**時，強調最小權限、生命週期管理與全程稽核，通知對象為**法定代理人**。
